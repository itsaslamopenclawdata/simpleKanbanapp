## Learning Objectives

The app exists to teach these fundamentals through direct practice:

- **HTTP verbs:** GET, POST, PUT, DELETE map directly to CRUD operations
- **SQL queries:** SELECT, INSERT, UPDATE, DELETE against a real PostgreSQL database with no ORM
- **Frontend-backend communication:** fetch API, JSON serialization, request/response cycle
- **DOM manipulation:** createElement, innerHTML, event listeners with vanilla JavaScript
- **Connection pooling:** pg library patterns that transfer to production

The "whoa" moment: adding a task, watching it appear in "To Do", changing the dropdown to "Done", and knowing exactly which HTTP request, SQL query, and DOM update made it happen.

**source:** Docs/Spec.md

## Key Design Philosophy

- Simplicity IS the feature
- Every line of code maps to a concept — no magic, no "it just works"
- The "strictly exclude" list is longer than the feature list
- Refetch-after-mutation pattern: the round-trip IS the lesson
- No optimistic updates, no client-side state cache
- Hardcoded credentials acceptable because it is a throwaway practice app on localhost

**source:** Docs/Spec.md

## Setup Sequence

1. Create PostgreSQL database: `createdb kanban_practice`
2. Create tasks table via psql
3. Initialize Node.js project: `npm init -y && npm install express pg`
4. Write `server.js` with pg connection and 4 CRUD endpoints
5. Write `index.html` with 3-column layout, task cards, and fetch-based CRUD
6. Run `node server.js`, open `http://localhost:3000`

## Frontend Layout Details

- Top: text input + "Add Task" button
- Below: three columns side by side — "To Do" | "In Progress" | "Done"
- Each task card: title, `<select>` dropdown (todo / in_progress / done), "Delete" button
- On page load: `fetch('/tasks')` renders cards into correct columns
- All mutations trigger full refetch + re-render
- Styling: basic inline CSS, flexbox or grid for columns, minimal card styling, no design system
