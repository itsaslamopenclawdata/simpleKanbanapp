## Problem Statement

Build a minimal Kanban board for local practice. The goal is learning HTTP verbs, SQL queries, and frontend-backend communication through the simplest possible working app. No frameworks, no build tools, no abstractions — just Express, pg, and vanilla HTML.

The prior design (FlowBoard) was a full production Kanban with 10+ models, real-time collaboration, auth, drag-and-drop, and Next.js. This is the opposite: one table, four endpoints, three files. The learning happens in the plumbing, not the features.

## What Makes This Cool

The simplicity IS the feature. You type `node server.js`, open `index.html`, and you have a working Kanban board. Every line of code maps to a concept: HTTP verbs, SQL queries, JSON serialization, DOM manipulation, fetch API. No magic. No "it just works." You see exactly how a web app fits together.

The "whoa" moment: adding a task, watching it appear in the "To Do" column, dragging the dropdown to "Done," and knowing exactly which HTTP request, SQL query, and DOM update made it happen.

## Constraints

- **Stack locked:** Node.js + Express + `pg` library, single `index.html` with inline CSS + vanilla JS
- **Single PostgreSQL table:** `tasks` with 4 columns
- **Four API endpoints max:** GET /tasks, POST /tasks, PUT /tasks/:id, DELETE /tasks/:id
- **Hardcoded DB credentials:** `host=localhost`, `port=5432`, `user=postgres`, `password=postgres`, `database=kanban_practice`
- **Zero production features:** No auth, no validation, no error handling, no logging, no pagination, no CORS configuration (allow all origins)
- **No external dependencies** beyond `express` and `pg`
- **No drag-and-drop, no animations, no frameworks, no build tools**

## Premises

1. **Single-table CRUD teaches fundamentals.** HTTP verbs map directly to SQL operations. No ORM, no abstraction layer. You write the query, you see the result.
2. **PostgreSQL is correct despite setup cost.** Learning real SQL against a real database is more valuable than SQLite's zero-config convenience. The `pg` library exposes connection pooling and query patterns that transfer to production.
3. **Vanilla HTML/CSS/JS is the right frontend.** Forces understanding of `fetch()`, DOM manipulation (`createElement`, `innerHTML`), and event listeners. No React, no virtual DOM, no state management — just the browser APIs.
4. **Refetch-after-mutation is fine.** Every add/update/delete triggers a fresh `GET /tasks`. No optimistic updates, no client-side state cache. The round-trip IS the lesson.
5. **Hardcoded credentials + zero error handling is acceptable.** This is a throwaway practice app that runs on localhost. Production patterns would obscure the fundamentals being learned.

## Approaches Considered

### Approach A: Node.js + Express + pg (chosen)
Three files. Bare-metal HTTP. You see every request, response, status code, and SQL query.
- Effort: S
- Risk: Low

### Approach B: Python + FastAPI + psycopg2
Auto-generated Swagger docs, type hints, cleaner async patterns. But FastAPI's abstractions hide the HTTP plumbing being learned.
- Effort: S
- Risk: Low

### Approach C: Express with minimal structure (4 files)
Split server.js into routes + db connection + migration script. Teaches separation of concerns. Slightly more files but same concepts.

## Recommended Approach

**Approach A: Node.js + Express + pg.** Three files. The most direct path to learning HTTP fundamentals. Express shows you exactly what happens with each request — no decorators, no dependency injection, no magic. You write `app.get('/tasks', ...)`, you see the SQL query, you return the JSON. That's it.

## Database

### Connection
- Host: `localhost`
- Port: `5432`
- User: `postgres`
- Password: `postgres`
- Database: `kanban_practice`

### Schema
```sql
CREATE TABLE tasks (
  id SERIAL PRIMARY KEY,
  title VARCHAR(255) NOT NULL,
  status VARCHAR(20) DEFAULT 'todo',
  created_at TIMESTAMP DEFAULT NOW()
);
```

### Allowed statuses
- `'todo'`
- `'in_progress'`
- `'done'`

## API Endpoints

| Method | Path | Request Body | Response | Description |
|--------|------|-------------|----------|-------------|
| GET | `/tasks` | — | `{ tasks: [...] }` | Returns all tasks |
| POST | `/tasks` | `{ "title": "string" }` | `{ task: {...} }` | Creates a task (status defaults to `'todo'`) |
| PUT | `/tasks/:id` | `{ "status": "string" }` | `{ task: {...} }` | Updates task status |
| DELETE | `/tasks/:id` | — | `{ success: true }` | Deletes a task |

All endpoints return JSON. `server.js` serves `index.html` via `express.static(__dirname)` so the frontend is served at `GET /` on the same origin as the API. No CORS configuration needed.

## Frontend

### Layout
- **Top:** Text input + "Add Task" button
- **Below:** Three columns side by side: "To Do" | "In Progress" | "Done"

### Task Card
Each card shows:
- Task title
- `<select>` dropdown with three options: `todo`, `in_progress`, `done`
- "Delete" button

### Behavior
1. On page load: `fetch('/tasks')` → render cards into correct columns
2. "Add Task" click: `fetch('/tasks', { method: 'POST', body: ... })` → refetch all → re-render
3. Dropdown change: `fetch('/tasks/:id', { method: 'PUT', body: ... })` → refetch all → re-render
4. "Delete" click: `fetch('/tasks/:id', { method: 'DELETE' })` → refetch all → re-render

### Styling
Basic inline CSS only. Three-column layout (flexbox or grid). Minimal card styling. No color palette, no design system.

## Setup Commands

```bash
# (a) Create database and table
createdb kanban_practice
psql -d kanban_practice -c "CREATE TABLE tasks (id SERIAL PRIMARY KEY, title VARCHAR(255) NOT NULL, status VARCHAR(20) DEFAULT 'todo', created_at TIMESTAMP DEFAULT NOW());"

# (b) Install dependencies
npm init -y
npm install express pg

# (c) Start the app
node server.js
```

Then open `http://localhost:3000` in a browser. Express serves `index.html` as a static file.
